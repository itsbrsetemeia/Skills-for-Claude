# Objective

Create and improve technical documentation that is clear, organized, useful, and easy to maintain.

The goal is not merely to write documentation.

The goal is to transfer knowledge efficiently.

# When to Use

Use this Skill when:

* The user needs to document a project.
* The user needs to create a README.
* The user needs to document an API.
* The user needs to create technical documentation.
* The user needs to document software architecture.
* The user needs to create user manuals.
* The user needs to create developer guides.
* The user wants to improve existing documentation.

# When NOT to Use

Do not use this Skill when:

* The user only wants to write code.
* The user wants to design software architecture.
* The user only wants grammar correction.
* The user is asking questions that do not require documentation.

# Philosophy

Good documentation answers questions before they are asked.

All documentation should be:

* Clear.
* Accurate.
* Up-to-date.
* Organized.
* Easy to navigate.
* Easy to maintain.

# Principles

Prioritize:

## Clarity

Explain concepts in a simple and objective manner.

Avoid unnecessary complexity.

## Accuracy

Never invent information.

When uncertainty exists, clearly state it.

## Consistency

Maintain:

* Consistent terminology.
* Consistent structure.
* Consistent formatting.

## Usefulness

Documentation should solve real problems for the reader.

# Documentation Types

Adapt the response according to the goal.

## README Files

Include:

* Project description.
* Features.
* Installation.
* Configuration.
* Usage.
* Contribution guidelines.
* License information.

## API Documentation

Document:

* Endpoints.
* HTTP methods.
* Parameters.
* Headers.
* Request examples.
* Response examples.
* Error codes.

## Libraries and Packages

Document:

* Installation.
* Configuration.
* Functions.
* Classes.
* Practical examples.

## Architecture Documentation

Explain:

* Components.
* Data flows.
* Dependencies.
* Technologies.
* Architectural decisions.

## Guides and Tutorials

Create:

* Tutorials.
* Step-by-step instructions.
* Best practices.
* Troubleshooting guides.

# Target Audience

Adapt the technical level for:

## Beginners

Explain fundamental concepts.

## Developers

Focus on implementation details.

## Technical Teams

Focus on maintenance and evolution.

## Stakeholders

Focus on objectives, value, and impact.

# Structure

Always organize content using:

* Headings.
* Subheadings.
* Lists.
* Examples.
* Tables when appropriate.

Avoid large blocks of unstructured text.

# Examples

Whenever possible, provide:

* Usage examples.
* Configuration examples.
* Code examples.
* Common use cases.

# Markdown Standards

Prioritize Markdown documentation.

Use properly:

* Headers.
* Lists.
* Tables.
* Code blocks.
* Links.
* Quotes.

# Review Process

When reviewing documentation, verify:

* Clarity.
* Technical accuracy.
* Grammar and spelling.
* Missing information.
* Structure.
* Navigation.

# Maintainability

Create documentation that can be easily updated.

Avoid:

* Redundant information.
* Confusing structures.
* Excessively long explanations.

# Response Format

When appropriate, organize responses as:

## Overview

General description.

## Objective

Purpose of the system or document.

## Installation

Setup instructions.

## Configuration

Required configuration steps.

## Usage

How to use the system or feature.

## Examples

Practical examples.

## Reference

Technical reference material.

## Troubleshooting

Common issues and solutions.

## FAQ

Frequently asked questions.

# Communication

Write in a manner that is:

* Professional.
* Objective.
* Organized.
* Educational.

Adjust complexity according to the audience.

# Language Adaptation

Always respond in the same language used by the user.

Examples:

* If the user writes in Portuguese, respond in Portuguese.
* If the user writes in English, respond in English.
* If the user writes in Spanish, respond in Spanish.

# Supreme Rule

The best documentation allows someone to use, maintain, or extend a system without needing to ask the original author any questions.
