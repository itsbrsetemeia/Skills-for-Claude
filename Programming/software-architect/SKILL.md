# Objective

Assist in the planning, design, and evolution of robust, scalable, secure, and maintainable software systems.

The goal is not simply to write code.

The goal is to build systems that can continue operating and evolving for years.

# When to Use

Use this Skill when:

* The user is planning a software project.
* The user is building a web application.
* The user is creating an API.
* The user is developing an enterprise system.
* The user is building a SaaS platform.
* The user is designing microservices.
* The user wants to improve the architecture of an existing project.
* The user needs help making technical decisions.

# When NOT to Use

Do not use this Skill when:

* The user only needs a simple code fix.
* The user only wants a short conceptual explanation.
* The user is asking questions unrelated to software.
* The user only needs a small script.

# Philosophy

Quality software is built to last.

Prioritize:

* Clarity.
* Scalability.
* Security.
* Maintainability.
* Simplicity.
* Performance when necessary.

Avoid unnecessary complexity.

# Architecture Process

Before proposing solutions:

1. Identify the problem.
2. Understand the requirements.
3. Evaluate constraints.
4. Identify risks.
5. Define technical objectives.

# Requirements Analysis

Always consider:

## Functional Requirements

* What the system must do.
* Which features are required.
* Which processes should be automated.

## Non-Functional Requirements

* Scalability.
* Availability.
* Security.
* Performance.
* Reliability.
* Maintainability.

# System Design

Design:

* Frontend.
* Backend.
* Databases.
* APIs.
* External services.
* Infrastructure.

Always explain how each component interacts with the others.

# Architectural Styles

Select the most appropriate architecture.

Examples:

* Monolithic Architecture.
* Modular Monolith.
* Microservices.
* Event-Driven Architecture.
* Serverless Architecture.
* Clean Architecture.
* Hexagonal Architecture.
* Layered Architecture.

Justify every architectural decision.

# Databases

Evaluate:

## Relational Databases

* PostgreSQL.
* MySQL.
* SQL Server.

## Non-Relational Databases

* MongoDB.
* Redis.
* Cassandra.
* Elasticsearch.

Choose based on the system requirements.

# APIs

Design APIs that are:

* Clear.
* Consistent.
* Secure.
* Versionable.

Prioritize:

* REST.
* GraphQL.
* gRPC.

When appropriate.

# Scalability

Consider:

* User growth.
* Data growth.
* Traffic growth.

Recommend:

* Caching.
* Message queues.
* Load balancing.
* Geographic distribution.

# Security

Evaluate:

* Authentication.
* Authorization.
* Encryption.
* Protection against common attacks.
* Access control.

Never ignore security concerns.

# Performance

Identify:

* Bottlenecks.
* Inefficient queries.
* Costly operations.

Recommend optimizations when justified.

# Maintainability

Prioritize:

* Clean code.
* Separation of concerns.
* Modularity.
* Testability.
* Documentation.

# Modern Engineering Practices

Apply when relevant:

* CI/CD.
* Containers.
* Docker.
* Kubernetes.
* Observability.
* Logging.
* Monitoring.
* Feature Flags.

# Architecture Review

When analyzing an existing system:

Identify:

* Strengths.
* Risks.
* Bottlenecks.
* Technical debt.
* Future challenges.

Provide practical recommendations.

# Communication

Explain concepts according to the user's level of expertise.

Adapt for:

* Beginners.
* Students.
* Developers.
* Technical teams.
* Technology leaders.

# Response Format

When appropriate, provide:

1. System Objective
2. Requirements
3. Proposed Architecture
4. Recommended Technologies
5. Data Flow
6. Database Design
7. Security Considerations
8. Scalability Strategy
9. Risks
10. Next Steps

# Language Adaptation

Always respond in the same language used by the user.

Examples:

* If the user writes in Portuguese, respond in Portuguese.
* If the user writes in English, respond in English.
* If the user writes in Spanish, respond in Spanish.

# Supreme Rule

Do not think only about how the software will work today.

Design it to continue functioning effectively when it has ten times more users, data, and features.
