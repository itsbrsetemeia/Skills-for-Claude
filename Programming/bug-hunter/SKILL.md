# Objective

Identify, investigate, and resolve bugs through a systematic, efficient, and evidence-based approach.

The goal is not merely to fix symptoms.

The goal is to uncover the root cause of the problem.

# When to Use

Use this Skill when:

* The user is experiencing a bug.
* The user receives error messages.
* The user encounters unexpected behavior.
* The user needs help debugging code.
* The user needs to investigate production failures.
* The user wants to analyze logs.
* The user needs to determine the source of an issue.

# When NOT to Use

Do not use this Skill when:

* The user wants to design a complete software architecture.
* The user is asking only for theoretical explanations.
* The user only wants code generation without analysis.
* The issue is unrelated to software.

# Philosophy

Every bug has a cause.

Never assume solutions before conducting an investigation.

Avoid:

* Premature assumptions.
* Superficial fixes.
* Changes without evidence.

Base conclusions on observable facts.

# Investigation Process

Always follow:

1. Understand the problem.
2. Reproduce the issue.
3. Gather evidence.
4. Isolate the cause.
5. Validate the hypothesis.
6. Implement the fix.
7. Verify the solution.

# Information Gathering

Always collect:

* Error messages.
* Logs.
* Code snippets.
* Execution environment details.
* Steps to reproduce.
* Dependency versions.
* Relevant configurations.

Never assume missing information.

# Error Analysis

Investigate:

## Syntax Errors

* Invalid characters.
* Incorrect structures.
* Type-related issues.

## Runtime Errors

* Exceptions.
* Startup failures.
* Missing resources.

## Logical Errors

* Incorrect conditions.
* Unexpected outcomes.
* Broken business rules.

## Integration Errors

* APIs.
* Databases.
* External services.
* Distributed systems.

# Root Cause Analysis

Always seek answers to:

* What happened?
* Why did it happen?
* When did it start?
* What changed?
* How can it be prevented in the future?

# Log Analysis

When reviewing logs:

* Identify relevant events.
* Separate symptoms from causes.
* Look for patterns.
* Correlate events chronologically.

Do not focus solely on the first error message.

# Performance Issues

Investigate:

* Slow execution.
* Excessive memory consumption.
* High CPU usage.
* Slow database queries.
* Inefficient loops.
* System bottlenecks.

# Security Issues

Identify vulnerabilities that may lead to:

* Data leaks.
* Privilege escalation.
* Injection attacks.
* Authentication failures.
* Exposure of sensitive information.

# Environments

Consider differences between:

* Development.
* Testing.
* Staging.
* Production.

Verify environment-specific configurations.

# Debugging Strategies

Use when appropriate:

* Logging.
* Breakpoints.
* Monitoring.
* Profiling.
* Isolated testing.
* Controlled reproduction.

# Hypothesis Validation

During investigation:

1. Form hypotheses.
2. Test hypotheses.
3. Eliminate incorrect hypotheses.
4. Confirm the correct hypothesis.

Avoid conclusions without validation.

# Response Format

Organize analyses as follows:

## Problem Summary

Clear description of the issue.

## Possible Causes

Prioritized list based on likelihood.

## Evidence

Relevant findings and indicators.

## Diagnosis

Explanation of the most probable root cause.

## Recommended Fix

Steps required to resolve the issue.

## Prevention

Measures to prevent recurrence.

# Communication

Adapt explanations for:

* Beginners.
* Students.
* Experienced developers.
* Technical teams.

Explain the reasoning process whenever possible.

# Language Adaptation

Always respond in the same language used by the user.

Examples:

* If the user writes in Portuguese, respond in Portuguese.
* If the user writes in English, respond in English.
* If the user writes in Spanish, respond in Spanish.

# Supreme Rule

Never stop at the first plausible explanation.

Continue investigating until the most probable root cause is supported by evidence.
